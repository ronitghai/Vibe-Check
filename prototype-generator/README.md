# AI Game Chat

A game library, not a chat window. Browse and instantly play 7 built-in games, or tap
"✨ Generate a game" to open a chat drawer and ask the LLM (via Groq) to customize a template or
generate a brand-new game from scratch. Every game — built-in or AI-generated — shares one dark
design system, so the library doesn't look like a pile of unrelated pages.

## Setup

**Backend**

```
cd backend
venv\Scripts\activate
copy .env.example .env    # then edit .env and set GROQ_API_KEY
uvicorn app.main:app --reload
```

Runs at http://localhost:8000. Requires `node` on PATH (used by
`backend/app/codegen/validator.py` to syntax-check generated JavaScript before serving it).

**Frontend**

```
cd frontend
npm run dev
```

Runs at http://localhost:5173.

## How it works

- `GET /api/library/{session_id}` — the 7 built-in templates (always available) plus any games
  generated by chat in this session.
- `POST /api/games/launch` — instantly launches a template with its default content, no LLM call.
- `GET /api/games/{session_id}/{game_id}` — returns one specific game as a fully assembled HTML
  document (template + config, or the validated generated bundle), with the shared design system
  and a strict CSP injected either way.
- `POST /api/chat` — the chat drawer's endpoint. Runs a Groq tool-calling loop: the model chats
  normally, calls `launch_template_game` (picks one of 7 games, writes custom content for it), or
  calls `generate_custom_game` (writes a whole new game) for anything else. Either tool call adds
  to the session's game collection — it shows up in the library from then on.
- The frontend mounts every game in a sandboxed `<iframe sandbox="allow-scripts" srcDoc=...>` —
  no `allow-same-origin`, no network access, isolated from the rest of the page.
- Session id is a UUID stored in `localStorage`; chat history and every launched game persist
  server-side in `backend/sessions.db` (SQLite) per session — reloading the page returns to the
  Library, which is itself the persistence story now.

## Design system

`backend/app/games/bundle.py` injects a shared `<style>` block (CSS custom properties for the
palette + `.btn`/`.game-title`/`.game-meta`/`.game-status` utility classes) into every game
bundle, template or generated, right after the CSP meta tag. The 7 templates in
`backend/app/games/library/*.html` are built on these tokens instead of hardcoded colors, and the
codegen system prompt (`backend/app/codegen/generator.py`) instructs the model to do the same.
`frontend/src/index.css` defines the identical palette for the outer app chrome, so the library
grid and chat drawer match the games themselves.

## Codegen safety

`backend/app/codegen/validator.py` runs generated HTML through a denylist (no network calls, no
external resources, no `document.cookie`/`localStorage`/`top`/`parent`) and a real JavaScript
syntax check (`node --check`) before it's ever stored or served. On failure the generator retries
once with the concrete error fed back to the model; if that also fails, the chat falls back to
suggesting a template game instead.

## Template game library

Tic-Tac-Toe, Wheel of Fortune, Flappy Bird, Memory Match, Matching Game, Crossword, Rapid Quiz —
`backend/app/games/library/`. Each is a single self-contained HTML file that reads
`window.__CONFIG__` for its content.
